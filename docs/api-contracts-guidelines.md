# API Contracts Guidelines

## 1. Source of Truth: Upstream Protobuf Definitions

All API contracts originate from the **upstream BSR module** `buf.build/project-kessel/inventory-api`. This repository does not contain `.proto` source files. The protobuf definitions live upstream and are consumed via `buf generate`.

- **RULE**: Never create or modify `.proto` files in this repository.
- **RULE**: To change an API contract, submit changes to the upstream `inventory-api` repository, then regenerate.

## 2. Code Generation Workflow

### 2.1 buf.gen.yaml Configuration

The file `buf.gen.yaml` at the repository root controls all code generation:

```yaml
version: v2
inputs:
  - module: buf.build/project-kessel/inventory-api
plugins:
  - remote: buf.build/protocolbuffers/pyi:v31.1
    out: src
    include_imports: true
  - remote: buf.build/protocolbuffers/python:v31.1
    out: src
    include_imports: true
  - remote: buf.build/grpc/python:v1.73.1
    out: src
    include_imports: true
```

- All three plugins use `include_imports: true` so that `buf.validate` protos are compiled locally (they are not available on PyPI).
- All three plugins use `out: src` to direct output to the `src/` directory -- all generated code lands under `src/kessel/inventory/`, `src/buf/`, and `src/google/`.
- When updating plugin versions, update all three plugins together to avoid version skew between `_pb2.py`, `_pb2_grpc.py`, and `.pyi` files.

### 2.2 Running Code Generation

```bash
buf generate
```

This single command downloads the latest protobuf definitions from BSR and regenerates all Python files. Run this as part of every release when upstream API changes exist.

### 2.3 Generated Files -- Do Not Edit

All files matching these patterns are generated and must never be manually edited:

- `*_pb2.py` -- Protobuf message classes
- `*_pb2_grpc.py` -- gRPC service stubs and servicers
- `*.pyi` -- Type stub files for IDE support
- Files under `src/buf/` and `src/google/` -- Imported dependency protos

Each generated file contains the header: `# Generated by the protocol buffer compiler.  DO NOT EDIT!`

Exclude generated files from linting and formatting. The `.flake8` config excludes `*_pb2*`; `black` commands must use `--exclude '.*_pb2(_grpc)?\.py'`.

## 3. API Versioning Strategy

### 3.1 Version Hierarchy

The repository contains three API version tiers under `src/kessel/inventory/`:

| Version | Stability | Purpose |
|---------|-----------|---------|
| `v1` | Stable | Health checks only (`GetLivez`, `GetReadyz`) |
| `v1beta1` | Deprecated | Legacy resource-specific services (K8s clusters, RHEL hosts, policies) |
| `v1beta2` | Current | Unified `KesselInventoryService` with generic resource model |

- New SDK consumers must use `v1beta2`. Do not build new features on `v1beta1`.
- `v1beta1` is retained for backward compatibility but uses a per-resource-type service design that has been superseded by the generic model in `v1beta2`.

### 3.2 Architectural Evolution: v1beta1 to v1beta2

The `v1beta1` API used **resource-specific services** (e.g., `KesselK8sClusterService` with `CreateK8sCluster`/`UpdateK8sCluster`/`DeleteK8sCluster`).

The `v1beta2` API consolidated everything into a single **`KesselInventoryService`** with a generic resource model (`ReportResource`/`DeleteResource`) where resource type is a string field, not a separate service.

When adding support for a new resource type, use `ReportResource` with the appropriate `type` field value. Do not create a new service definition.

## 4. Request/Response Type Conventions

### 4.1 Naming Pattern

Every RPC method has a dedicated request and response message, each in its own file:

| Method | Request | Response |
|--------|---------|----------|
| `Check` | `CheckRequest` | `CheckResponse` |
| `CheckSelf` | `CheckSelfRequest` | `CheckSelfResponse` |
| `CheckForUpdate` | `CheckForUpdateRequest` | `CheckForUpdateResponse` |
| `CheckForUpdateBulk` | `CheckForUpdateBulkRequest` | `CheckForUpdateBulkResponse` |
| `CheckBulk` | `CheckBulkRequest` | `CheckBulkResponse` |
| `CheckSelfBulk` | `CheckSelfBulkRequest` | `CheckSelfBulkResponse` |
| `ReportResource` | `ReportResourceRequest` | `ReportResourceResponse` |
| `DeleteResource` | `DeleteResourceRequest` | `DeleteResourceResponse` |
| `StreamedListObjects` | `StreamedListObjectsRequest` | `StreamedListObjectsResponse` |
| `StreamedListSubjects` | `StreamedListSubjectsRequest` | `StreamedListSubjectsResponse` |

- Each request/response type lives in a separate `_pb2.py` file named after the type in snake_case. Follow this convention for any new types.
- Bulk variants use an `Item` message (e.g., `CheckBulkRequestItem`) wrapped in a top-level request with an `items` repeated field.

### 4.2 Shared Reference Types

Core reusable types in `v1beta2`:

- `ResourceReference` -- identifies a resource (resource_type, resource_id, reporter)
- `SubjectReference` -- identifies a subject (wraps ResourceReference, optional relation)
- `ReporterReference` -- identifies a reporter system (type field)
- `Consistency` -- oneof: minimize_latency, at_least_as_fresh (token), at_least_as_acknowledged
- `RequestPagination` / `ResponsePagination` -- cursor-based pagination with continuation_token
- `WriteVisibility` -- enum: UNSPECIFIED, MINIMIZE_LATENCY, IMMEDIATE
- `Allowed` -- enum: UNSPECIFIED, TRUE, FALSE

### 4.3 Field Validation

Fields use `buf.validate` annotations (not manually enforced):
- Required message fields: `[(buf.validate.field).required = true]`
- Required strings: `[(buf.validate.field).string.min_len = 1]`
- Pattern-constrained strings: regex like `^[A-Za-z0-9_-]+$` for type/reporter_type fields

## 5. Hand-Written SDK Code

### 5.1 Allowed Locations for Manual Code

Only these files contain hand-written code and may be edited:

- `src/kessel/inventory/__init__.py` -- `ClientBuilder` base class and `client_builder_for_stub()` factory
- `src/kessel/inventory/v1/__init__.py` -- `ClientBuilder` for health service
- `src/kessel/inventory/v1beta2/__init__.py` -- `ClientBuilder` for inventory service
- `src/kessel/auth/auth.py` -- OAuth2 credential management
- `src/kessel/auth/__init__.py` -- Public auth exports
- `src/kessel/grpc/__init__.py` -- `oauth2_call_credentials()` adapter
- `src/kessel/rbac/v2/__init__.py` -- RBAC convenience functions
- `src/kessel/middleware/__init__.py` -- Middleware (currently empty)
- `examples/*.py` -- Example scripts
- `tests/*.py` -- Unit tests

### 5.2 ClientBuilder Pattern

Each API version exposes a `ClientBuilder` via its `__init__.py`:

```python
from kessel.inventory import client_builder_for_stub
from kessel.inventory.v1beta2.inventory_service_pb2_grpc import KesselInventoryServiceStub
ClientBuilder = client_builder_for_stub(KesselInventoryServiceStub)
```

When adding a new API version, create an `__init__.py` that exposes a `ClientBuilder` using `client_builder_for_stub()` with the appropriate stub class.

## 6. gRPC Method Patterns

### 6.1 Unary Methods

Most methods are `unary_unary` (single request, single response): `Check`, `CheckSelf`, `CheckForUpdate`, `CheckForUpdateBulk`, `CheckBulk`, `CheckSelfBulk`, `ReportResource`, `DeleteResource`.

### 6.2 Server-Streaming Methods

List operations use `unary_stream` (single request, streamed response): `StreamedListObjects`, `StreamedListSubjects`.

- Client code must iterate over the stream. For sync: `for response in stub.StreamedListObjects(request)`. For async: `async for response in stub.StreamedListObjects(request)`.
- Implement pagination by reading `response.pagination.continuation_token` and re-issuing requests with `RequestPagination(limit=N, continuation_token=token)` until the token is empty.

## 7. RBAC v2 REST Layer

The `kessel.rbac.v2` module provides REST-based workspace operations via the RBAC v2 API:

- Endpoint pattern: `GET {rbac_base_endpoint}/api/rbac/v2/workspaces/?type={root|default}`
- Required header: `x-rh-rbac-org-id`
- Returns JSON with `data` array containing workspace objects
- RBAC workspace fetching uses REST (requests library), not gRPC. This is a separate service from the inventory gRPC API.

## 8. Adding a New API Version

To add support for a new upstream API version (e.g., `v1beta3` or `v2`):

1. Ensure the new version exists in the upstream `buf.build/project-kessel/inventory-api` module.
2. Run `buf generate` -- new files will appear under `src/kessel/inventory/{new_version}/`.
3. Create `src/kessel/inventory/{new_version}/__init__.py` exposing a `ClientBuilder`.
4. Add convenience functions in `src/kessel/rbac/{new_version}/__init__.py` if needed.
5. Add examples under `examples/`.
6. Add tests under `tests/`.
7. Update `README.md` to document the new version.

## 9. Backward Compatibility Rules

- Never remove a version directory. Old versions must remain importable for existing consumers.
- The SDK follows SemVer. Removing an API version requires a major version bump.
- SDK versions across languages (Python, Ruby, Go) are independent and do not need to be synchronized.
- Pin `protobuf` dependency to a compatible range (currently `>=6.31.1,<6.34.0` in `pyproject.toml`). Protobuf runtime version must match the version used in code generation.

## 10. Protobuf Runtime Compatibility

Each generated `_pb2.py` file contains a runtime version check:

```python
_runtime_version.ValidateProtobufRuntimeVersion(
    _runtime_version.Domain.PUBLIC, 6, 31, 1, '', ...)
```

The `protobuf` dependency version in `pyproject.toml` must be compatible with the version used by the buf plugins in `buf.gen.yaml`. If you update plugin versions, update the protobuf dependency range accordingly.
